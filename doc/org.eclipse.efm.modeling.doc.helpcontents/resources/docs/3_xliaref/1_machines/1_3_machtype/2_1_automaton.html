Statemachine as automata

Definition and declaration

This kind of statemachine models simple automata. The submachines of an automaton are machines of type state.

Those "state" objects are declared within the "@region:" section :

Model of Computation

An automaton may have several states but they being active is controlled. Therefore, the statemachine is declared with a "moc" either "or" or "xor".

Difference between "or" and "xor" :

  • "moc: or", one or several states can be active simultaneously
  • "moc : xor", one and only one state can be active simultaneously (we recommend that all automata be modeled as "xor" statemachines which is the only formally correct model)

Initial state

Initializing an "automaton" statemachine requires to specify which one is/are its starting states. Indeed, the engine needs to know at which state the machine must be at the beginning of the simulation

There are two ways of doing so in XLIA. Those two ways use "state flags".

  • We strongly recommend using this method over the next one, as it reproduces the standard UML design.
  • This is the UML-compliant method because like for most graphical modeling languages, in UML, initial states are fundamentally different from other states.
  • The advantage of this method is that we can do initialization operations (such as setting variable values...) at the beginning by completing the "instruction block" of the "startup" transition.

The Quick method

With this method there is no need to have a third state. We just add the flag "start" to the "A1" state so that the "A" state-machine will go to this state at the beginning of the simulation.

Remarks :

  • The "start" state is an syntactic shortcut for both the only initial state and the simple state in which the statemachine will be after its initialization, i.e. after the evaluation of its initial transition.
